Harness engineering

Not a prompt trick. A managed software loop.

Xolotl treats coding agents like production systems: explicit inputs, bounded tools, observable state, repeatable checks, and durable memory.

Context 01

Read the codebase before acting

Graph structure, local instructions, design systems, and recent git state become first-class context before the model writes code.

Control 02

Route work to the right model

Use Kimi for coding-heavy loops, MiniMax for huge codebase passes, GLM or Qwen for balanced tasks, and frontier models when needed.

Verification 03

Green CLI before green commit

Each loop ends with format, tests, lint, smoke commands, and a commit that records a verified step forward.

Model-agnostic by design

One harness, many execution profiles.

Xolotl normalizes provider quirks into the same operator surface: connect the key, choose the model, run the loop, inspect the result.

Operating principles

Make the agent earn every edit.

01

Surface assumptions

Ambiguity is tracked before implementation, not hidden in code.

02

Keep changes surgical

Every touched line should trace back to the current objective.

03

Cache expensive knowledge

Graph reports, design files, and prior outcomes reduce token spend.

04

Commit verified increments

Shipping means the CLI is green and the repository state is durable.

Field manual

Build agents like systems, then let the system improve itself.

Xolotl is the harness layer for long-running coding work: model selection, context discipline, tool execution, testing, commits, and iterative learning in one operator loop.
